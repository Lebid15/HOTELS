# SECURITY — ملاحظات أمان منصّة funduqii

## نموذج التهديد المختصر
منصّة SaaS متعدّدة المستأجرين (فنادق) ببيانات نزلاء ومدفوعات. التهديدات الرئيسية:
اختراق حدود المستأجرين، تسريب PII للنزلاء، تلاعب مالي، إساءة الحجز العام، سرقة الجلسة.

## المصادقة والجلسات
- JWT (SimpleJWT) + Refresh Token مع **تدوير** وقائمة سوداء عند الخروج.
- **قفل الدخول:** بعد 5 محاولات فاشلة/15 دقيقة (cache).
- **التحقّق بخطوتين (2FA)** اختياري لكل مستخدم — قناة «الكود داخل النظام» (يظهر للمدير)؛ قنوات خارجية (Email/SMS/TOTP) نقطة توسّع.
- **موثوقية الجلسة:** اعتراض عام (`SessionGuard`) يجدّد التوكن عند 401 ويعيد الطلب، وعند الفشل يُنهي الجلسة برسالة واضحة (لا فشل صامت).
- **تخزين التوكن:** `localStorage`/`sessionStorage` (قرار حالي). المخاطرة: تعرّض لـXSS. **مخفَّف** بغياب `dangerouslySetInnerHTML` كليًّا. نقطة توسّع: httpOnly cookies.

## الأدوار والصلاحيات
- ثلاثة أدوار: `platform_owner` / `manager` / `reception` عبر `UserProfile.role`.
- **fail‑closed (م6):** الدور يُقرأ **حصريًّا** من `UserProfile.role` — لا استنتاج من `username`/`email`. مستخدم بلا Profile ⇒ **بلا دور** (لا وصول). أُزيل احتياطي «اسم المستخدم → دور» القديم. (نظام أدوار الـAPI لا يعتمد `is_superuser`؛ وصول admin منفصل.)
- `BaseHotelResourcePermission`: تصنيف حسب الدور + حجب الفندق الموقوف + `reception_methods` لكل مورد.
- **عزل المستأجرين:** كل `get_queryset` يصفّي بـ`hotel_id` المشتقّ من المستخدم؛ حقل `hotel` للقراءة فقط في الـSerializers.
- صلاحيات الموظف الدقيقة تُعرَض للواجهة؛ **إلزامها الشامل خادميًّا** نقطة تحسين مجدولة.

## سلامة المال
- **لا حذف نهائي لسجلّ مالي** — الدفعات ثابتة (get/post فقط)، والإبطال (`void`) بسبب إلزامي + Audit Log؛ يشمل **Payment** و**Expense** و**FolioCharge** (م1: لا DELETE، إبطال + استثناء المبطل من الرصيد).
- **اتساق الدفع المقسّم (م4/م8):** مجموع طرق الدفع = الإجمالي (Payment، وFoodOrder بأجزائه الأربعة بما فيها «على حساب الغرفة»)، بلا قيم سالبة — يُرفَض عدم التطابق خادميًّا (Decimal).
- المشتقّات (`paid`/`balance_due`) تُحسب من الصفوف غير الملغاة؛ منع الخروج بالدين مُلزَم خادميًّا (402).

## الحجز العام
- رقم حجز عشوائي غير قابل للتخمين (`WEB-YYYY-XXXXXX` عبر `secrets`).
- **رمز إدارة آمن** (`manage_token`) — الإدارة/الإلغاء لا يعتمدان على الهاتف وحده. **(م3)** يظهر الرمز و`manage_url` **مرّة واحدة فقط** في استجابة الإنشاء؛ لا يُكشفان في البحث/العرض (مُسلسِلان منفصلان).
- **تطبيع الهاتف (م5):** بحث/إدارة الحجز يعتمدان صيغةً مطبّعة (`guest_phone_normalized` مفهرس + backfill) فلا يفشل البحث لاختلاف صيغة الرقم؛ الحقل المطبّع **لا يُعرَض** في Public APIs.
- منع التصادم عبر `select_for_update` داخل معاملة ذرّية.
- بوّابة أهلية الفندق المركزية (`eligibility.py`: نشط + اشتراك صالح + باقة + اتفاقية) قبل قبول الحجز/الظهور — القرار كلّه خادميّ (م2).
- **Throttling** بـ scope مستقلّ لكل نقطة عامة حسّاسة (توفّر/إنشاء/بحث/إلغاء/تقييم) (م7).
- تقنيع الهاتف/البريد في الاستجابات العامة؛ رسائل فشل عامّة لا تكشف تفاصيل.

## إعدادات الإنتاج
- الإقلاع **يفشل** إن غاب `SECRET_KEY` مع `DEBUG=false`.
- HSTS/SSL‑redirect/Secure‑cookies تُفعَّل تلقائيًا بقيَم آمنة عند `DEBUG=false`، وقابلة للتجاوز عبر متغيّرات `DJANGO_SECURE_*` (انظر `DEPLOY.md §2.1` و`backend/.env.example`).
- **م9:** `python manage.py check --deploy` بمتغيّرات إنتاجية حقيقية ⇒ **0 issues** (الأمر الكامل في `DEPLOY.md §2.2`). `ALLOWED_HOSTS`/`CORS`/`CSRF` من env؛ `SECURE_PROXY_SSL_HEADER` مضبوط للعمل خلف بروكسي HTTPS.
- حارس بذر يرفض التشغيل على الإنتاج بلا `ALLOW_SEED=1`.

## سجلّ التدقيق (Audit Log)
سجلّ ثابت (append‑only) للأحداث الحسّاسة: دخول/خروج، مدفوعات (إنشاء/إبطال)، مصاريف (إبطال)، حالات الفندق، الاشتراكات، العمولات، الاتفاقية، أحداث الأمان (login_success/failed/locked/2fa)، إغلاق اليوم.

## الإبلاغ عن ثغرة
تُبلَّغ الثغرات إلى مسؤول المنصّة عبر البريد الرسمي، مع مهلة إفصاح مسؤول قبل النشر العلني.

## نقاط توسّع أمنية (غير مانعة، مؤجّلة/مجدولة)
- **Object Storage للملفات/الوسائط** (S3/R2) بدل التخزين المحلّي عند التوسّع.
- **CSP أقوى لاحقًا** (إزالة `unsafe-inline`) بعد **تقليل inline styles** تدريجيًّا.
- **استراتيجية جلسة/توكن** (httpOnly cookies بدل localStorage) + إدارة جلسات/أجهزة + خروج من كل الأجهزة.
- **قنوات 2FA خارجية** (Email/SMS/TOTP).
- **مراجعة تقييمات قبل النشر** (`is_approved` افتراضيًّا).
- معالجة ثغرات `npm audit` المتوسطة (ترقية تبعيات مخطّطة).
- **اختبار اختراق خفيف** قبل التوسّع.
- ضبط `NUM_PROXIES`/`X‑Forwarded‑For` على الإنتاج خلف بروكسي (ليعمل throttling على IP الحقيقي).
