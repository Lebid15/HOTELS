## 1) الرحلة السعيدة — العابرة للأدوار (Golden Path)

اختباران متكاملان يغطّيان السلسلة الكاملة من طرف إلى طرف:

### أ) رحلة الموقع العام — `EndToEndScenarioTests.test_public_website_journey_end_to_end`
| # | الخطوة | التحقّق |
|---|--------|---------|
| 1 | مالك المنصّة يؤهّل الفندق (نشط + اشتراك فعّال + باقة تسمح + إظهار) | إعداد صريح |
| 2 | الفندق يظهر في الموقع العام (بلا مصادقة) | `GET /api/public/hotels/` يحوي الفندق |
| 3 | نزيل يحجز من الموقع | `201` + رقم `WEB-…` + `manage_token` آمن |
| 4 | العمولة تُشتقّ لحظة الحجز (لقطة لمالك المنصّة) | `BookingCommission` موجودة |
| 5 | المدير يرى الحجز ضمن فندقه بمصدر `public_website` | `GET /api/reservations/{id}/` |
| 6 | دخول → دفع جزئي → **منع الخروج بالدين (402)** → تسوية وخروج | `check_in`→`payment`→`check_out=402`→`settle_and_checkout` |
| 7 | العمولة **تبقى** بعد الخروج (لا حذف) + سجلّ التدقيق يعكس السلسلة | `BookingCommission` باقية + `reservation.settle_checkout` |
| 8 | إغلاق اليوم يعكس مدفوعات الرحلة | `payments_total > 0` |

### ب) رحلة الحجز المباشر (استقبال/مدير) — `EndToEndScenarioTests.test_full_operational_chain`
حجز مباشر يُؤكَّد تلقائيًا → دخول → دفعة جزئية → طعام على الغرفة + رسم فوليو →
**الدين المشتقّ = 120** → **منع الخروج (402)** → تسوية ذرّية وخروج → الغرفة → تنظيف →
سجلّ التدقيق (`check_in`/`payment.create`/`settle_checkout`) → إغلاق اليوم مخزّن.
